# Design decisions

Synthesis of Phase A discovery (ux-portfolio-architect, portfolio-story-editor, brand-art-director), reconciled into one direction. Where the three disagreed, this section is the tie-break — implementation follows this file, not the raw agent output.

## 2026-07-11 (fourteenth pivot): bio cube full-bleed photos, header logo glow, About page rework, connect-button redesign, header/hero background seam fix

Five independent asks, mostly refinement on top of the thirteenth pivot's ship.

**Bio cube, second pass** (`BioCard.tsx`): `FACE_HEIGHT` 640→720 (a bit taller). Each `ExperienceFace`'s photo switched from a `h-1/2` top-half crop to `absolute inset-0` (full face height) — the text section (title + description) is now an overlay near the bottom, `rgba(8,8,10,0.82)` + `backdrop-filter: blur(10px)` ("barely transparent," not the previous half-height opaque split). The age/country signage doubled in size (font/padding, not a CSS `scale()` — deliberately avoided, per this project's own repeated-bug history of `scale()` compounding with other transforms in the same chain) and moved from straddling the top edge (`translate(-50%,-50%)`) to `bottom-full` with a margin gap, so it now stands fully above the face like a sign mounted on the block's roof, not half-buried in the photo.

**Header logo button**: `profile.handle` corrected `"@mrtarasyuks"` → `"@mrtarasyuk"` (typo fix, matches the already-correct `profile.telegram` handle). The button itself extracted from `SiteHeader`'s generic `HeaderPillLink` into a new client island, `HeaderLogoButton.tsx` (`SiteHeader` stays a Server Component; this follows the exact same pattern as `WorldSwitchHeaderNav`/`ThemeToggle` beside it) — it reads the live selected world from `WorldNavContext` (mounted globally in the root layout, so this works on every route, not just the homepage where the world switcher is interactive) and glows in that world's signal color. Enlarged per the ask, but only from `sm:` up (`sm:px-8 sm:py-3 sm:text-lg`) — a literal 2x at every breakpoint pushed the mobile-only icon row (theme/language/menu, `md:hidden`) off-screen at 320px, caught by `responsive.spec.ts`'s overflow test, not a guess: measured via `document.documentElement.scrollWidth` before and after, same "measure, don't guess" approach this file has documented before for header collisions.

**About page rework** (`about/page.tsx`): the left-column `t.about.eyebrow` label ("ABOUT"/"ПРО МЕНЕ") removed — homepage `AboutSection.tsx`'s own teaser keeps its own eyebrow, this was scoped to the dedicated page only, so the copy key itself stays. `AnimatedName` (plain fade+rise `<h1>`) replaced by `WorldTitleCube` (the same hover-tilt glass cube used for `/work`'s title) — `AnimatedName.tsx` deleted outright, zero remaining references. Body paragraphs were typing in parallel (`startDelayMs={300}` on every paragraph, independent of each other) — fixed with a cumulative-delay `reduce` (each paragraph's start delay is the previous paragraph's start delay plus its own length × speed) so the whole body types through sequentially, start to finish, as one continuous reveal. Hit the React Compiler's purity rule shipping this: an initial version mutated a captured `let cumulativeDelayMs` inside `.map()`, which `react-hooks/immutability` rejects even in a Server Component with no hooks (the rule applies to any component function, not just ones using hooks) — fixed with a `.reduce()` that threads the running delay through a fresh array each step instead of reassigning an outer variable.

**Connect button redesign** (`ConnectBurstButton.tsx`): the `size="lg"` button (About page only — `size="md"` has no other call site) widened (`px-14`, `min-w-[280px]`) and gained a continuous idle shake+bounce (`animate-cta-bounce`, repurposing the previously dead `float-name`/`animate-float-name` keyframe — zero real usages existed, same shape almost exactly fit, renamed rather than left as unused CSS) at a 2s cycle, running only while the burst menu is closed. The open animation changed from "links slide down as a vertical list" to "three thin strips grow down from the button first (staggered 90ms), then the links fade in already arranged in a row" — the links' `flex-col` container became `flex-row`, and their reveal delay was pushed back to start only once the strips finish growing. A throwaway verification script's first click attempt timed out — Playwright's actionability check waits for an element to stop moving before clicking, and the new continuous bounce animation never settles; real users aren't held to that "stable" requirement, so this is a test-tool quirk (worked around with `.click({ force: true })` in the script), not a product bug.

**Header/hero background seam in light theme, found and fixed**: reported as "a strip at the top, invisible in dark theme" — diagnosed (not guessed) by sampling the actual painted background color at several y-coordinates via `elementFromPoint`/`getComputedStyle`, which showed the header's own padding area (transparent) revealing the plain body `--bg`, while the `.world-scope` section directly below it (homepage hero, world gallery pages) paints a different per-world pale `--w-bg` in light theme — a real seam at the header/section boundary, invisible in dark theme only because both colors happen to be near-black there. Fixed by giving `.world-scope` itself `margin-top: calc(-1 * var(--header-h, 0px))` plus equal `padding-top` (the same negative-margin-then-pad-back trick the hero's inner canvas div already used, now applied to the section's own box so its background paints from y=0 behind the transparent header, while its content stays at the same visual position via the matching padding). Since this class is shared by both the homepage hero and the `/work/[world]` gallery pages, `<HeaderHeightVar />` (previously mounted only inside `HeroOrbitScene`) was moved to the root layout so `--header-h` is reliably published on every route, not just the homepage.

Seven independent asks in one round, introducing two small reusable patterns applied repeatedly rather than one-off solutions per component: an **embossed/volumetric label** (the `PageTitleWatermark` hollow-outline-plus-glow technique, reused at smaller scale for corner watermarks and the bio cube's "signage") and a **magnetic tilt-on-hover** (the pointer-relative `perspective()/rotateX()/rotateY()` idiom `WorldChooserBlocks` already used, now built into `WorldTitleCube` itself so every usage gets it for free).

**Homepage drum carousel, real glass**: `DrumProjectCard`/`DrumComingSoonCard` (the latter converted from a plain dashed div to a real `GlassPanel`, for consistency across the whole reel) now use a new `--glass-tint-strong` token (dark `rgba(255,255,255,0.14)`, light `rgba(10,10,11,0.09)`, alongside the existing `--glass-tint` pair) instead of the airier default — passed via `style` on `GlassPanel`, which reliably out-ranks the class the same way `GlassProjectBlock`'s fixed background already does. `GlassPanel` also gained an opt-in `edgeDistortion` prop: an `feTurbulence`/`feDisplacementMap` SVG filter (id scoped per-instance via `useId()`, same duplicate-id reasoning as `FlagIcon`) applied only within a radial rim mask plus a touch of extra blur, so refraction reads at the edges while the content on top stays crisp. `backdrop-filter: url(#...)` support is inconsistent across browsers — this is a progressive enhancement that silently no-ops where unsupported, not a functional dependency.

**Bio cube rebuilt as 4 uniform "experience" faces** (`BioCard.tsx`): the old asymmetric layout (front = full bio text, right/back/left = bare photo) is gone, replaced by one shared `ExperienceFace` rendered on all four sides via the existing `CubeFace` wrapper (front now goes through it too, at `rotateDeg={0}`) — Project Manager (front, reusing `BIO_CARD_PORTRAIT_SRC`), Middle AI Developer (right), Video Creator (back), 3D Modelling + Print (left), each with a short truthful description grounded in already-verified project facts (Foody's Gemini/Claude stack, the AI Video Production pipeline, the 3D Lab). Every face is now a fixed `h-full`/`h-1/2` split (photo top half, bold title + description bottom half) rather than content-driven flow — the cube wrapper carries one explicit `FACE_HEIGHT` (640px) instead of the front face's natural height setting it, so all four faces match regardless of each description's length; `HALF_DEPTH` (half of `CARD_WIDTH`, still 540) is untouched since that relationship only constrains the horizontal rotation footprint, not height. The photo slot switched from the old single-portrait `aspect-[659/781] bg-contain` (uncropped, tuned for one true headshot) to `aspect`-free `h-1/2 bg-cover` — an intentional supersession of that earlier rule now that every face is a generic experience card. A small "signage" pill straddles the top edge of each face (`translateZ(26px)` to stand proud, layered `textShadow` for an embossed look) showing age + `profile.location`, identically on all four faces rather than one non-rotating 3D-anchored element. Copy's `bioCard.role`/`ageLabel`/`countryLabel`/`hobbiesLabel`/`hobbies` (now unused) were removed outright, not left dead; `bioCard.experiences: {title, description}[4]` replaces them (en+uk, `CopyDict` shape enforced).

**Hero floor grid, thicker + brighter**: each of the three backdrops' `FloorGrid` opacity multiplier raised ~55–60% (`ServerColumnField`/`FrameCardField` `×0.14`→`×0.22`, `PrintRingField` `×0.16`→`×0.24`) for brightness. For thickness, `GridHelper` can't reliably take a real line width cross-browser (native WebGL `linewidth` is ignored by ANGLE/most GPUs) — rather than pull in `Line2`/fat-lines machinery (already rejected for the spotlight beam in an earlier pivot as too heavy for a decorative effect), `FloorGrid.tsx` now renders two extra faint `gridHelper` copies, diagonally jittered by ±0.035 units and additively blended (`depthWrite: false`), reading as a soft thicker/glowing line. Contained entirely in this one file — no caller API changes, all three worlds get it automatically.

**Developers' lamp lines thicker**: `ServerColumnField.tsx`'s lamp `cylinderGeometry` args `[0.02, 0.02, pos.h, 6]` → `[0.05, 0.05, pos.h, 8]` (radius 2.5×, one more radial segment so the now-visibly-cylindrical lamp reads smooth). Only the lamps — the pulse sphere and debris untouched.

**`/work` index redesign**: the three `WorldChooserBlocks` grew (`h-56/h-64` → `h-72/h-80`), gained a bigger glyph and a large embossed corner watermark of the world's own name (bottom-right, same technique as `PageTitleWatermark` but scoped to the block) so each of the three reads as a distinct large "signature" instead of three identically-sized generic rectangles; the tagline switched from `text-text-muted` to `text-text` — already a theme-reactive pair resolving to near-white in dark mode and near-black in light mode, exactly the ask, no new CSS needed. The page's plain `<h1>Work</h1>` became a centered `WorldTitleCube` (which gained an optional `glyph` — skipped when absent — and a `headingTag` prop defaulting to `h1` so a second reuse can render `h2` instead, keeping one real heading per page); because `WorldTitleCube`'s hover-tilt is now built into the shared component, `WorldGallery.tsx`'s existing per-world titles picked up the same interaction for free, covering the "add mouse-move animation to gallery section titles" ask with no separate edit. `SelectedWorkList`'s heading got the identical `WorldTitleCube` treatment (new `accent` prop, defaulting to `worldThemes.developers.signal`) — and its copy changed from "Selected work"/"Обрані роботи" to "My work"/"Мої роботи" per direct request. The dead gap between the three blocks and the list below was trimmed (`mt-20/mt-28` → `mt-12/mt-16`).

**Product blocks overhaul** (`ProjectCardFlagship.tsx`, new `ProjectCardCompact.tsx` parity, new shared `LogoBadge.tsx`): each product row is now a glowing `GlassPanel` card tinted by the project's own `getWorldTheme(project.world).signal` — ties "vivid/beautiful" to something meaningful even though this list itself is world-agnostic chrome — plus the same embossed corner-watermark motif (project name) used above. A new `projectLogoSrc(slug)` helper (`assetPaths.ts`) is existence-checked server-side in `SelectedWorkList.tsx` (identical `publicAssetExists` pattern to `hasPortrait`) and passed down as `hasLogo`; `LogoBadge` renders the real file once one exists, a lit world-glyph placeholder until then. The case-study link is now a real solid CTA pill (project's signal color fill) instead of a plain underlined link; any populated `project.links` (none yet) render as additional outline pills beside it. The stack list — previously plain uppercase text — is now a row of glowing rounded-full chips (border + soft glow + gradient fill in the project's signal color). Picked up two small pre-existing gaps while rewriting this exact code: the "Role"/"Stack" labels were hardcoded English strings with no locale variant — added `selectedWork.roleLabel`/`stackLabel` (en+uk) instead of leaving the hardcode in place.

## 2026-07-11 (twelfth pivot): cube corner cleanup, bigger equalizers, real 3D floor grid in light theme, mouse-tilt hover on world blocks

Quick follow-up round, four items.

**BioCard cube**: the dashed borders on every face (`2px dashed ${color}` on the front face and all three photo faces) are removed outright — plain clean edges. Anchor moved further left (`BIO_CARD_ANCHOR` x: `-3.15` → `-3.6`).

**Equalizer bars made chunkier**: `[0.16, 0.08, 0.16]` → `[0.28, 0.17, 0.28]` (`BAR_SIZE`/`BAR_HEIGHT` constants), with `BAR_GAP` widened `0.11` → `0.2` to keep the taller bars from intersecting — same bar count, same overall stack height ratio, just each segment reads as a noticeably more voluminous block.

**Light-theme hero grid — corrected to recolor the real 3D floor, not layer a second flat one.** The eleventh pivot's flat CSS graph-paper pattern (`.world-light-surface`'s `background-image`) technically satisfied "pale background + grid," but on the hero it sat as a static 2D pattern behind a 3D scene that already has its own moving, perspective-converging `FloorGrid` mesh — reading as two disconnected grids, not "the same floor, recolored," which is what was actually asked for. Fixed by giving `FloorGrid.tsx` a `theme` prop that swaps its `GridHelper`'s line color to near-black (`#0a0a0b`) in light theme (recreating the `gridHelper` via a memoized `args` tuple — R3F disposes and reconstructs a primitive when its `args` change by value, which is the only way to change a `GridHelper`'s baked-in vertex colors after construction) and removing the flat CSS pattern from the hero entirely — it now lives only on `.world-gallery-page` (a class reintroduced specifically for the `/work/[world]` pages, which have no 3D floor to match). `theme` is threaded down as a plain prop through `HeroOrbitScene` (reads `useTheme()` itself, outside the `<Canvas>`) → `WorldBackdrop` → each backdrop field (`ServerColumnField`/`PrintRingField`/`FrameCardField`) → `FloorGrid`, the same prop-drilling pattern this chain already used for `scrollSpeed`/`paused`/`world`, deliberately not a `useTheme()` call from inside a Canvas descendant (consistent with why those other three were already threaded as props instead of read via context hooks in place). Hit a naming collision shipping this: `WorldBackdrop.tsx` already had a local `const theme = getWorldTheme(world)` (the per-world palette object) — the new prop is destructured as `theme: siteTheme` there specifically to avoid shadowing it.

**World-chooser blocks (`/work` index), hover redone as a center-anchored tilt.** The eleventh pivot's `group-hover:-translate-y-1 group-hover:scale-[1.03]` lifted the block and moved it on screen — not what was asked for ("moves in place, doesn't get displaced"). Replaced with a `perspective()/rotateX()/rotateY()` tilt that follows the pointer's position within the block, applied via `onPointerMove`/`onPointerLeave` mutating a ref'd wrapper's `style.transform` directly (not React state — same "no re-render on a continuous pointer-driven update" reasoning as the bio cube's own drag and the drum carousel), computed relative to the block's own center so it only ever rotates in place, never translates. The signal-colored glow shadow stays a plain CSS `group-hover:shadow-[...]` (an unrelated property, no conflict).

**Heads up for later**: a Hetzner server and a domain are already purchased for eventually deploying this site — not yet acted on, no deployment work done this round.

## 2026-07-11 (eleventh pivot): cube geometry/opacity fix, real cube photos, /work transition + gallery-header redesign, cube-shaped equalizer bars, light theme reaches the hero canvas

Follow-up correction round on the tenth pivot's ship, six items.

**BioCard cube was reading elongated — a real compounding-transform bug, not a proportions guess.** The rAF loop wrote `perspective(2200px) rotateY(deg) scale(0.5)` as one transform on one node every frame. Chaining `scale()` after `perspective()` in the same transform list shrinks the effective perspective distance the cube's own children see (their `translateZ()` values are unscaled local pixels, but the perspective origin establishing "how strong is the foreshortening" is itself computed in the now-half-scale local space) — same *class* of bug as the `distanceFactor`/width compounding note from the ninth pivot, just one call deeper. Fixed by splitting into two nested wrappers: an outer node that only ever holds `scale(0.5)` (set once, static), and an inner node the rAF loop mutates with only `perspective(2200px) rotateY(deg)`. Also switched from a responsive `520/580px` width pair to one fixed `CARD_WIDTH = 540` — a 3D-anchored Html element's on-screen size is already governed by distanceFactor/scale, not viewport width, so the breakpoint variance served no purpose and actively fought the cube's `HALF_DEPTH = CARD_WIDTH / 2` needing to equal exactly half the width for a square (not elongated) footprint. **Cube faces made genuinely opaque**: the front face's `bg-white/[0.06] backdrop-blur-xl` (translucent glass) and the placeholder photo faces' `to-black/50` gradient both let the 3D scene bleed through; replaced with solid dark backgrounds. **Text pinned to fixed white/bold tones** (`text-white`, `font-bold`) instead of the `text-text`/`text-text-dim`/`text-text-muted` theme tokens it inherited before — those tokens flip toward dark grays in light theme, which combined with the now-opaque-dark card would have made the text unreadable; the card lives inside the theme-invariant hero canvas, so its text needs to be theme-invariant too, same reasoning BioCard's colors already followed pre-cube. **Real photos wired**: the user dropped three files directly into `public/img/` (`cube-img-card01.png/02.jpg/03.jpg`) — `assetPaths.ts`'s three placeholder constants pointed at the original guessed filenames and were updated to match what was actually supplied, no other code changes needed (the existence-check pattern already handled the rest).

**`/work` page fly-transition coverage bug, found and fixed**: the clicked block's fly-forward animation relied on the other two blocks fading to `opacity: 0`, which should have hidden them — but the flying block itself had no explicit `z-index`, so plain DOM paint order (later siblings on top) decided what covered what. Clicking the *first* or *second* block let a later, still-fading sibling visually paint over it mid-transition. Fixed with `zIndex: isTarget ? 40 : 0` on the button wrapper, so the flying block always paints above its siblings regardless of click order or DOM position.

**`/work` gallery pages, header redesign**: the tenth pivot's `"{WORLD}. WORKS"` title is gone (`orbit.worksLabel` copy key removed outright, not left dead) — replaced with a `WorldTitleCube` (new, `components/work/WorldTitleCube.tsx`) styled as a small volumetric glass cube: a folded right-spine + bottom-edge (BioCard's original pre-cube-rewrite fake-3D technique, reused here since a static "display plaque" doesn't need full drag interactivity) plus a soft blurred radial-gradient glow behind it in the world's signal color, reading as backlit from inside the glass. A round glass "back" button (new `orbit.back` copy key) sits beside it, linking to `/work`. **Hover feedback added to the `/work` world-chooser blocks** (`WorldChooserBlocks.tsx`): scale-up + a signal-colored glow shadow on `group-hover`, applied to the inner `GlassPanel` rather than the outer button — the outer button already carries an inline `transform` for the fly/mount animation, and inline styles always beat a Tailwind class for the same CSS property (the same lesson as the `--w-bg` cascade bug two pivots back), so a competing `group-hover:scale-*` class on that same element would silently never apply.

**`GlassProjectBlock` redesign**: bigger card (`min-h-[300px]`, `p-8`), title moved into the same top row as the status badge/year but right-aligned in large bold white text, description text switched from the dim `text-text-muted` token to a fixed `text-white/80`. The card's background is forced to a fixed dark `rgba(9,9,11,0.68)` via `style` (not the shared `GlassPanel`'s own theme-reactive `--glass-tint` class) specifically so the white text stays legible regardless of site theme — same fixed-dark-card-needs-fixed-light-text pairing as the BioCard fix above, applied to a component that previously used theme tokens correctly (Work pages are meant to theme) but wasn't going to any more once its background stopped being theme-reactive. Hit a real wrapping bug shipping this: the status badge (e.g. "SYSTEM DESIGN") squeezed into a shrinking flex column once a long title claimed most of the row's width and wrapped onto two lines; fixed with `shrink-0 whitespace-nowrap` on the badge cluster so the *title* wraps first, not the metadata.

**Equalizer bars now read as cubes, not flat plates**: `boxGeometry` args changed from `[0.5, 0.08, 0.02]` (a thin flat card) to `[0.16, 0.08, 0.16]` (equal width/depth) — same bar count and vertical spacing as before (the "tall as they are now" ask), just each individual segment is chunky/cube-shaped from any viewing angle instead of a plate that only reads as a bar face-on.

**Light theme now reaches the hero canvas's background too** — a real scope change from the tenth pivot's decision, not a bug. The pale-background-plus-black-grid treatment (renamed `.world-gallery-page` → `.world-light-surface`, since it's no longer gallery-page-only) is now applied to `HeroOrbitScene.tsx`'s section alongside the `/work/[world]` pages. This does **not** touch the actual WebGL scene (avatar materials, fog color, backdrop particles all stay exactly as dark/atmospheric as before) — only the DOM layer sitting behind the canvas's transparent (`alpha: true`) background changes, which shows through wherever the 3D scene has no geometry rendered. Reasoning for reversing the tenth pivot's "hero canvas is theme-invariant" scoping: repeated user feedback made clear "the world's color" was understood to include the homepage hero, not just its separate gallery page, and the DOM-layer-only version of this change is low-risk (verified: the drum carousel's project cards, which already used theme-reactive tokens correctly, read fine against the new pale backdrop with no further changes needed).

## 2026-07-11 (tenth pivot): bio cube, multicolor backdrops, light-theme world pages, Developer/Video content split

Ten more numbered items on top of the ninth pivot, touching the hero canvas, the light theme, and the content model.

**BioCard → a real draggable cube** (`src/components/scene/BioCard.tsx`, rewritten): the old single-face card plus two fake "folded spine" divs is now a genuine 4-sided CSS cube — front face is the existing bio content (portrait, name, role, age/base, hobbies), the other three faces (right/back/left, `rotateY(90/180/-90deg) translateZ(HALF_DEPTH)`) are plain photo panels. Horizontal drag spins it around Y via the same "mutate a ref every rAF tick, decaying flick velocity on release" idiom already used by the avatar and the homepage drum — chosen over React state so a 60fps drag never forces a re-render. Native Pointer Capture (`setPointerCapture`) is used here (unlike the avatar's window-listener workaround), since this is a plain DOM element, not an R3F raycast target, so there's no capture ambiguity to sidestep. The card is also **half its previous size**: rather than halving every literal padding/font-size inside, a single `scale(0.5)` was folded into the existing `perspective()/rotateY()` transform chain — the same "one CSS operation, not a pixel-budget rewrite" logic the ninth pivot's `distanceFactor` note already established. The role line now runs through `TypewriterText` (the "fills in like typed text" ask) rather than static text. The three new photo faces are wired through the same `hasPortrait`-style optional-asset pattern (`BIO_CARD_PHOTO_RIGHT/BACK/LEFT_SRC` in `assetPaths.ts`, existence-checked server-side in `page.tsx`, threaded through `HeroSceneGate` → `HeroOrbitScene` → `BioCard`) — a quiet gradient placeholder renders per face until a real photo is dropped in, zero further code needed.

**Multicolor backdrops, two places.** (1) *Developers' lamp-fade lines* (`ServerColumnField.tsx`): previously lerped between the world's signal yellow and one warm-white tone; now each line cycles independently through a full HSL hue rotation (`Color.setHSL`, phase-offset per line same as before) so grid-scroll-on reads as genuinely multicolored blinking, not a two-tone fade. (2) *3D's pulsing ring "vases"* (`PrintRingField.tsx`): replaced outright with `EqualizerBars` — a fixed vertical stack of thin boxes per anchor, each with a **static** color drawn from a green→red spectrum (classic VU-meter look) and only its *opacity* flipping between lit/dim as a shared "level" value rises and falls on a randomized cadence, LED-meter style. Chosen over animating each bar's actual scale (cheaper, no geometry mutation, matches this codebase's existing preference for opacity-driven "fake" motion over transform recomputation) — same easing-back-to-flat-on-toggle-off behavior the old vases had.

**Developers' floating cubes now spin and drift for real** (`FloatingDebris.tsx`, new `GravityCubes` export used only by `ServerColumnField`): the drei `Float`-wrapped box debris (gentle wobble only) is replaced by a component that mutates `rotation.x/y` continuously every frame (real per-axis spin, the "rotates around its own axis" ask) plus a slow sine/cosine position drift (the "floating in zero gravity" read) — reuses the existing `DebrisSpec[]` array (its `speed`/`rotX`/`rotY` fields repurposed as rotation speed and drift-phase seeds) rather than a second randomized generator. 3D's octahedron debris and Video's `FallingDonuts` are untouched — this was specifically about the Developers-world boxes called out by name.

**Light-theme pale backgrounds + black grid on the `/work/[world]` gallery pages** (`globals.css`): these three HTML pages previously hardcoded each world's always-dark `theme.bg` regardless of the light/dark toggle (a mismatch with this project's own rule that content pages, unlike the hero canvas, theme with the toggle). Added `.world-gallery-page` — a class scoped to `WorldGallery.tsx`'s wrapper only, deliberately distinct from the `.world-scope` class the homepage hero section also carries, so the hero canvas (which stays per-world-dark regardless of theme, by design, per the eighth pivot) is completely unaffected by this change. Pale gray for Developer, pale turquoise for Video Creator, pale beige for 3D, each with a faint black graph-paper grid overlay. Hit a real cascade bug shipping this: the pale `--w-bg` override was silently losing to the *same* custom property set inline (`style={vars}`) on the *same element* in `WorldGallery.tsx` — inline styles always out-rank an external stylesheet rule for the same property, no selector specificity can beat that, only `!important` can; fixed by marking the three override declarations `!important`.

**Content model split**: the AI Video Production System project moved from `world: "video"` to `world: "developers"` (`developerCategory: "app"`) — it's a software system/pipeline, not video footage, so it now shows as a Developer-world case study instead. `WorldGallery.tsx`'s Video-world branch no longer renders a generic project grid at all (there are no video-world projects left) — it goes straight to the existing `ToolsUsed`/`VideosByCategory` sections, matching the ask that Video Creator show only genres/styles/categories, with actual clips landing inside those categories later. `tests/projects.spec.ts` updated to match: Developer world now asserts both Foody and the AI Video System card, Video Creator's drum carousel is asserted against its numbered placeholder slots instead of a real project title.

**Copy**: `orbit.worlds.developers` "Developers" → "Developer" (both locales — it was already an intentionally untranslated brand-style label, same as "Video Creator"). New `orbit.worksLabel` ("Works"/"Роботи") composes each gallery page's `<h1>` as `{WORLD}. {WORKS}` (e.g. "DEVELOPER. WORKS", "3D. WORKS"). About page's `about.body` (both locales) rewritten in first person, warmer and less list-like — same facts, no new claims, per this project's content-truth rules.

**Smaller fixes**: About page's video panel pulled up (`md:-mt-16 lg:-mt-24`) so it sits higher relative to the name/text column. The 3D world's camera station (`CameraRig.tsx`) pulled back and widened (`z: 8.3→9.6`, `fov: 38→41`, `lookAt.y: -0.35→-0.55`) — the avatar's head was reading as too close to the header at common viewport sizes; verified via the same throwaway-screenshot method as previous pivots.

**Real avatar-fit bug, not a Strict-Mode red herring**: user-reported symptom was the avatar rendering tiny and far away specifically in dev mode (never in the production build). Root cause, confirmed by instrumenting the actual bounding-box numbers rather than guessing: `Avatar.tsx`'s fit `useLayoutEffect` measures with `THREE.Box3.setFromObject()`, which **always measures in world space** and internally calls `object.updateWorldMatrix(false, false)` — it deliberately does not refresh ancestor matrices or force a recompute. Two distinct failure modes stacked on top of each other:
1. Immediately after `group.scale.setScalar(1)` resets the fit group (the existing Strict-Mode double-invoke mitigation, documented in the entry below), the very next `Box3().setFromObject(scene)` could still read `scene`'s *stale* `matrixWorld` from before the reset, because nothing forces a synchronous recompute — `Box3` just trusts whatever's cached. Fixed by calling `group.updateMatrixWorld(true)` after every scale/position mutation and before every measurement.
2. Even with matrices always fresh, `Box3.setFromObject` still measures in **world space**, and this fit group lives under `Avatar`'s `Float`-bobbing wrapper and idle-spin group — so if the effect happens to run while Float is mid-bob (verified: real, not cached-stale, Y offset — a pure vertical shift with the box's own span unchanged, which a Y-rotation could never produce, only an ancestor's Y-position could), that bob gets silently baked into the "how tall is this model" measurement and therefore into the computed floor position. Rotation was ruled out specifically because Y-axis rotation cannot change a Y-extent. Fixed by temporarily `parent.remove(group)` / `parent.add(group)` around the whole measure-and-position block — with no parent, `Box3` measures purely the model's own intrinsic geometry, immune to wherever its animated ancestors currently are. Verified deterministic across 6 repeated cold-load runs (previously the two Strict Mode passes disagreed by ~1.3 world units on the Y offset alone) via a throwaway instrumented build, output confirmed then removed — same disposable-debugging-script convention used elsewhere in this file.

**Avatar is bigger**: `TARGET_HEIGHT` 3 → 3.6 (a flat 20% increase, chosen as a reasonable first pass since "make it big" wasn't quantified — easy to retune). The platform pedestal, its rim ring, and the invisible drag hit-cylinder are all scaled by the same 1.2× so their proportions to the avatar stay exactly as designed, not just "still large enough."

## 2026-07-10 (later same day): real avatar + three-world atmosphere system

Second pivot on top of the orbit hero below, driven by explicit user feedback: the flat starfield+planet backdrop was "boring/static," and a real 3D avatar (`public/model/model-tarasiuk.glb`, Tripo3D-generated) was supplied to replace the wireframe placeholder. **Asset pipeline.** The source GLB was 16.8MB (267K vertices, uncompressed textures) — compressed via `gltf-transform optimize` (Draco geometry compression, meshopt simplification to ~35% of vertices, WebP textures at 1024px) down to 807KB, a ~20x reduction. Original kept at `assets-source/model-tarasiuk-original.glb` (gitignored, local only); the optimized file ships in `public/model/`. Draco decoder files copied from `node_modules/three/examples/jsm/libs/draco/gltf/` into `public/draco/` so decoding is self-hosted, not dependent on Google's CDN.

**Avatar integration** (`src/components/scene/Avatar.tsx`): `useGLTF` + `Suspense`, falling back to the original `AbstractFigure` wireframe while the model streams in (same component, zero new code for the fallback). Auto-fits any model regardless of native scale/pivot: measures the loaded scene's bounding box, scales to a target world-space height, then re-centers — critical detail, **the fit calculation resets the wrapping group's transform to identity before each measurement**, because React Strict Mode double-invokes effects in development and the first pass's applied scale would otherwise get compounded by the second pass measuring an already-scaled object (this exact bug produced a "changing the target height does nothing" symptom that took a debug-logging pass to catch — see the model's actual native height was ~0.98 units, and the second effect invocation was measuring the *already-rescaled* 6-unit-tall result and computing a corrective scale of 1, silently undoing the first pass). The base material/geometry never changes per world — only a `drei` `Outlines` rim tint and a nearby point light recolor. This is deliberate: relighting one constant subject reads as "one person, three moods"; changing materials per world would read as costume changes instead.

**Anti-fragmentation mechanism.** Three palettes is the easy part — the risk is three worlds reading as three unrelated microsites. Fix, per the art-director pass: identical typography/card grammar/orbit mechanic across all worlds; exactly one accent color per world, used sparingly (never a wash — small text, borders, one light, one rim, same restraint discipline signal-yellow already had); and **theming is scoped to only two zones** — the hero canvas + its HTML chrome while a tab is active, and that world's `/work/[world]` gallery page. Global nav, footer, and the below-the-fold sections (Across Layers, Capabilities, How I Work, About, Contact) stay in the original Obsidian Signal palette regardless of active tab — one constant frame, three windows opening inside it.

**Per-world tokens** live in `src/content/worldTheme.ts`. Developers keeps the original `globals.css` palette unchanged (dark obsidian + signal yellow — the color already established site-wide). 3D and Video Creator are *not* generic domain clichés (rejected: sci-fi cyan-grid for 3D, red-clapperboard for video) — each accent is drawn from the literal color of the tool involved: 3D's `#ff6a2c` is an FDM print-nozzle's heat glow; Video's `#2de0c2` is a broadcast-monitor/vectorscope phosphor tone, deliberately desaturated and used sparingly so it doesn't tip into neon-cyberpunk territory (a real risk the brief explicitly bans).

**Backdrops** (`src/components/scene/backdrops/`): the flat starfield+planet is gone entirely, replaced per world — `ServerColumnField` (Developers: fogged vertical light-columns with one traveling pulse, evoking unattended server infrastructure at night), `PrintRingField` (3D: a stack of thin torus "print-layer" cross-sections with a traveling nozzle-height plane), `FrameCardField` (Video: wireframe 16:9 storyboard frames with a traveling waveform/scrub line). Each uses `FogExp2` for real depth occlusion (stars never had this — it's what makes a starfield read as flat wallpaper vs. these reading as volumetric). Crossfade between worlds: `WorldBackdrop.tsx` keeps the outgoing and incoming field mounted simultaneously for ~0.6s, fading opacity in opposite directions — never a hard cut.

**Engineering note on the crossfade + purity rules.** The new `eslint-plugin-react-hooks` "purity" rules (React Compiler-oriented) reject `Math.random()` inside `useMemo`/`useRef` initializers (flagged as impure-during-render) and reject reading a ref's `.current` inside JSX/render (`react-hooks/refs`). Both showed up building this feature and needed real restructuring, not suppression: particle-field random positions moved to module-scope constants (computed once at import, not per-render); the backdrop crossfade moved from "parent computes opacity-by-ref every frame and passes it down as a prop" to "each field owns its own fade via `useFrame`, driven by stable `direction`/prop values rather than a per-frame ref read in the render body."

**CSS-side theming.** The HTML chrome around the hero (name/role text, world tabs, nav-button hover, CTA hover) reads scoped CSS custom properties (`--w-bg`, `--w-signal`, `--w-text-dim`) set via inline `style` on a wrapping `.world-scope` element, with plain `transition: color 700ms ease` rules in `globals.css` — changing a custom property's value while a `transition` is declared on the property that *consumes* it animates smoothly with zero JS/lerping needed. Only the WebGL-side color (fog, avatar rim/light) needs an actual per-frame `Color.lerp` (`src/hooks/useLerpedColor.ts`), since Three.js has no CSS-transition equivalent.

**Three world gallery pages.** `/work/3d`, `/work/video`, `/work/developers` — static named route segments (not a dynamic `[world]`, which would collide with the existing `/work/[slug]` dynamic segment at the same tree level; Next.js allows one static segment to coexist with one dynamic sibling, and static routes take routing priority). Cost: the three world names are now reserved and can never be used as a project slug — documented directly above the `worlds` export in `src/content/types.ts`. Each gallery page reuses the existing `ProjectCardFlagship`/`ProjectCardCompact` components filtered by world, wrapped in the same `.world-scope` CSS-variable theming as the hero (page background + role/tab colors shift per world; card internals intentionally keep the standard dark styling rather than re-theming every nested component — a scope boundary to keep the work tractable, not an oversight).

**Verification.** No visual/browser tool available by default in this environment — used a throwaway Playwright script (loaded, screenshotted all three worlds + mobile, read the images back) to actually see the result, which is how the avatar-scaling Strict-Mode bug above was caught (a `console.log` of the computed bbox during the same debugging pass showed the second effect invocation measuring the wrong thing). - Orbit ring sits on its own Y plane well below the figure's feet (`centerY` prop on `OrbitRing`), not at the figure's height — the front-facing card is always screen-X-centered same as the figure, so only a Y offset avoids permanent overlap. Getting this right required accounting for perspective: cards at the *front* of the orbit are much closer to the camera than the figure itself, so they have a **much tighter vertical frustum** than the figure does at its own depth — a large Y offset that clears the figure can push the front card below the visible frame entirely. Tuned camera distance/orbit radius/centerY together and verified visually (see below) rather than by formula alone.
- Category switcher lives in normal document flow *below* the canvas, not as an absolute overlay inside it — an in-canvas overlay collided with orbiting cards at some rotations; moving it out avoids fighting 3D staging for a piece of UI that doesn't need to be "in" the scene.
- Progressive enhancement, not a hard requirement: `HeroSceneGate` renders the original flat `HeroIdentity` (Capability Signal Rail) during SSR and first paint, and only swaps to the 3D scene client-side after confirming WebGL support via `useSyncExternalStore` (avoids hydration mismatches — server and first client render always agree). No WebGL, or JS disabled → flat hero, unchanged from before. This also keeps initial HTML crawlable/indexable.
- `prefers-reduced-motion` stops auto-rotation, the figure's continuous spin/bob, and the planet's rotation (`paused` prop threaded through `SpaceBackground`/`AbstractFigure`/`OrbitRing`), but user-initiated prev/next clicks still animate — an intentional, bounded exception, consistent with how `CapabilitySignalRail` already behaved.
